Provider Abstraction

src/providers/sarvam.py · elevenlabs_stt.py · whisper.py

Three ASR providers, three different APIs, three different auth patterns. The provider layer wraps each behind a unified transcribe() interface so the evaluation runner doesn't care which provider it's testing.

Each wrapper is ~30 lines. They all take the same arguments and return the same dict shape:

src/providers/sarvam.pySarvam API wrapper
def transcribe( audio_path: str, language_code: Optional[str] = None, model: str = "saaras:v3", ) -> dict: headers = {"api-subscription-key": SARVAM_API_KEY} with open(audio_path, "rb") as f: mime_type = mimetypes.guess_type(audio_path)[0] or "audio/wav" # Sarvam API rejects audio/mp4a-latm; use audio/mp4 for m4a files if mime_type == "audio/mp4a-latm": mime_type = "audio/mp4" files = {"file": (Path(audio_path).name, f, mime_type)} start = time.perf_counter() response = requests.post(API_URL, headers=headers, files=files, data=data) latency = time.perf_counter() - start return { "provider": "sarvam", "model": model, "transcript": result.get("transcript", ""), "language_code": result.get("language_code"), "latency_seconds": round(latency, 3), }
Design decision
Sarvam uses a custom api-subscription-key header instead of standard Bearer auth. ElevenLabs and OpenAI both use their official SDKs. The wrappers abstract this so the runner just calls provider.transcribe().

The three providers also differ in language code format. Sarvam uses BCP-47 (hi-IN), while ElevenLabs and OpenAI use ISO-639-1 (hi). This mapping lives in the evaluation runner, not the providers:

src/run_eval.pyLanguage code mapping
# Sarvam uses BCP-47 codes SARVAM_LANG_MAP = { "hi": "hi-IN", "ka": "kn-IN", "en-IN": "en-IN", "hi-en": "hi-IN", "kn-en": "kn-IN", } # ElevenLabs and Whisper use ISO-639-1 WHISPER_LANG_MAP = { "hi": "hi", "ka": "kn", "en-IN": "en", "hi-en": "hi", "kn-en": "kn", }
Gotcha: MIME type bug
The highlighted lines in the Sarvam wrapper are a workaround. Python's mimetypes.guess_type() returns audio/mp4a-latm for .m4a files, but Sarvam's API rejects this MIME type with a cryptic 400 error. We override it to audio/mp4. Without this fix, all personal banking recordings (recorded as .m4a) would fail.

Evaluation Runner

src/run_eval.py

The runner is the orchestrator. It loads ground truth, loops through every (audio file, provider) pair, computes metrics, and saves everything to CSV + JSON. It's a straightforward nested loop with error handling.

src/run_eval.pyMain evaluation loop
def run_evaluation(entries, providers, limit): results = [] for entry in entries: audio_path = entry["audio_path"] reference = entry["text"] # ground truth transcription for provider_name in providers: result = transcribe_single(provider_name, audio_path, language) if result["status"] == "ok": wer_result = compute_wer(reference, result["transcript"]) cer_result = compute_cer(reference, result["transcript"]) results.append({ "filename": filename, "provider": provider_name, "reference": reference, "hypothesis": result.get("transcript"), "wer": wer_result["wer"], "cer": cer_result["cer"], "latency_seconds": result.get("latency_seconds"), }) return results

Ground truth comes from two JSON files: dataset_ground_truth.json (14 IndicVoices + 7 Svarah files from public datasets) and personal_template.json (7 banking recordings I made myself). The loader merges them:

src/run_eval.pyGround truth loading
def load_ground_truth() -> list: entries = [] # Public dataset ground truth dataset_gt = GROUND_TRUTH_DIR / "dataset_ground_truth.json" if dataset_gt.exists(): with open(dataset_gt) as f: for item in json.load(f): item["audio_path"] = str(AUDIO_DIR / item["source"] / item["filename"]) entries.append(item) # Personal recordings ground truth personal_gt = GROUND_TRUTH_DIR / "personal_template.json" if personal_gt.exists(): with open(personal_gt) as f: data = json.load(f) for item in data.get("clips", data): item["source"] = "personal" entries.append(item) return entries
Why two data sources?
Public datasets (IndicVoices, Svarah) provide diverse speakers and accents but cover general topics like cooking and sports. Personal recordings are the only banking-specific test data in the evaluation. Combining them lets us compare "general Indic" vs "domain-specific code-mixed" performance.

Total dataset: 28 audio files × 3 providers = 84 API calls per full evaluation run. Each result includes WER, CER, latency, and the raw reference/hypothesis text for debugging.

Metrics Engine

src/metrics/wer.py · cer.py

WER and CER computation is delegated to jiwer, an open-source library purpose-built for speech recognition evaluation. Our wrappers add one critical step: text normalization before comparison.

src/metrics/wer.pyWER with normalization
def compute_wer(reference: str, hypothesis: str) -> dict: ref_norm = normalize_text(reference) hyp_norm = normalize_text(hypothesis) output = jiwer.process_words(ref_norm, hyp_norm) return { "wer": round(output.wer, 4), "substitutions": output.substitutions, "deletions": output.deletions, "insertions": output.insertions, }
Why CER matters for Indic languages
WER counts whole-word errors. For agglutinative languages like Kannada, a single character error (a nuqta, a vowel sign) makes the entire word "wrong" — even though 90% of the characters are correct. CER catches this distinction.

Example: Kannada transcription with 27% WER but only 13% CER = the errors are minor character differences, not complete word failures. WER alone would make you think the model is bad. CER tells you it's actually close.

Both compute_wer() and compute_cer() normalize text before comparison. Without this, legitimate transcription differences (punctuation, casing, Unicode variants) inflate error rates artificially.

Text Normalization

src/metrics/normalize.py

This is the most critical module in the pipeline. Without it, WER is meaningless for Indic languages. The normalization pipeline handles Unicode variants, script-specific characters, and punctuation across three writing systems.

src/metrics/normalize.pyThe pipeline (order matters)
def normalize_text(text: str) -> str: text = unicode_normalize(text) # 1. NFC normalization text = normalize_devanagari_variants(text) # 2. Nuqta + chandrabindu text = strip_punctuation(text) # 3. Latin + Devanagari + Kannada text = normalize_numbers(text) # 4. Comma stripping, rupee symbol text = text.lower() # 5. Case folding text = collapse_whitespace(text) # 6. Multiple spaces → one return text.strip()

The most impactful step is nuqta stripping. In Devanagari, a nuqta (़) is a dot placed under a consonant to indicate a borrowed sound. Different transcribers handle it differently:

src/metrics/normalize.pyDevanagari variant normalization
def normalize_devanagari_variants(text: str) -> str: # Strip nuqta (U+093C) — normalizes क़→क, ज़→ज, फ़→फ, etc. text = text.replace("\u093C", "") # Chandrabindu ँ → Anusvara ं (common transcription variation) text = text.replace("\u0901", "\u0902") return text
What breaks without this
Without nuqta stripping, the word हज़ार ("hazār", thousand) and हजार ("hajār") are treated as completely different words — inflating WER by ~50% on short texts that contain currency amounts. Both are valid transcriptions of the same spoken word.

Pipeline order matters. NFC normalization must run first because decomposed Unicode forms can split a nuqta into a separate combining character. Lowercasing runs after script normalization to avoid interfering with Devanagari (which has no case). Whitespace collapse runs last to clean up gaps left by punctuation stripping.

Script Normalization

src/metrics/script_normalize.py

This module exists because of a single discovery that changed the entire evaluation's conclusions.

The 40pp discovery
ElevenLabs scribe_v2 outputs English loanwords in Latin script (credit card) while ground truth uses native Devanagari (क्रेडिट कार्ड). This inflated code-mixed WER by 40 percentage points — from 56.61% (raw) to 17.01% (after script normalization). Without this correction, you'd wrongly conclude ElevenLabs can't handle code-mixing.

The solution: build Latin → native script maps from the ground truth data, then replace Latin words in the hypothesis before WER computation:

src/metrics/script_normalize.pyLatin → Devanagari mapping
# Derived from ground truth in personal_template.json _LATIN_TO_DEVANAGARI = { "credit": "क्रेडिट", "card": "कार्ड", "transaction": "ट्रांज़ैक्शन", "payment": "पेमेंट", "account": "अकाउंट", "fraud": "फ्रॉड", # ... 16 more banking terms }
src/metrics/script_normalize.pyThe normalization function
def normalize_script(text: str, language: str) -> str: if language == "hi-en": mapping = _LATIN_TO_DEVANAGARI elif language == "kn-en": mapping = _LATIN_TO_KANNADA else: return text # No normalization for monolingual words = text.split() normalized = [] for word in words: clean = re.sub(r"[.,!?;:]", "", word).lower() if is_latin_word(word) and clean in mapping: normalized.append(mapping[clean]) else: normalized.append(word) return " ".join(normalized)
Why this is applied selectively
Script normalization only runs on code-mixed files (hi-en, kn-en). Monolingual Hindi or Kannada files don't have Latin loanwords. Applying it everywhere would be a no-op at best, and could introduce false matches at worst.

The is_latin_word() check prevents false positives — a word is only considered Latin if >50% of its characters are ASCII letters. This avoids normalizing Devanagari words that happen to contain a stray Latin character.

Post-ASR Correction

src/correction/llm_correction.py · confidence_guided.py

Can you fix ASR errors after the fact with an LLM? This module is a proof-of-concept that tests two approaches: full-transcript correction (send everything to GPT) and targeted correction (only fix low-confidence words).

src/correction/llm_correction.pyBanking domain system prompt
BANKING_SYSTEM_PROMPT = ( "You are correcting ASR transcription errors from Indian banking calls.\n\n" "RULES:\n" "1. ONLY fix clear transcription errors.\n" "2. NEVER standardize dialect to written form.\n" "3. NEVER merge or split words across boundaries.\n" "4. Preserve the original script (Devanagari, Kannada, Latin).\n" "5. If unsure, LEAVE IT UNCHANGED.\n" "6. For code-mixed: English banking terms are strong signal.\n" "7. Numbers, amounts: preserve the format used by the speaker.\n\n" "Banking vocabulary: खाता, ब्याज, किस्त, राशि, जमा, निकासी, शेष, " "ಖಾತೆ, ಬಡ್ಡಿ, ಕಂತು, ಮೊತ್ತ.\n\n" "Return ONLY the corrected text." )

The confidence-guided pipeline for ElevenLabs uses per-word logprob scores to decide what to correct:

src/correction/confidence_guided.pyDecision tree
# Step 1: Re-transcribe with word-level confidence stt_result = transcribe_with_word_confidence(audio_path, seed=seed) # Step 2: Flag low-confidence words (logprob < -0.5) flagged = flag_low_confidence_words(words, threshold=-0.5) # Step 3: Choose correction strategy if len(flagged) >= MIN_FLAGGED_FOR_TARGETED: # Enough flagged words → targeted correction correction = correct_flagged_words(transcript, flagged, words) else: # Too few flagged → fallback to full-transcript correction = correct_full_transcript(transcript)
The calibration problem
ElevenLabs logprobs are near-zero even at 50–115% WER on Indian languages. The confidence scores are not calibrated for Indic content. Without the fallback (highlighted), the targeted correction becomes a no-op: fewer than 2 words get flagged per file, and the LLM never sees the errors.
Key finding: Model quality > Post-processing
Correction only helps weak models. Whisper-1 (43% WER) improved by 15pp with LLM correction. But Sarvam (15% WER) actually got worse — the LLM overcorrected dialectal variants to standard written forms. Upgrading from whisper-1 to gpt-4o-transcribe achieved the same 15pp improvement as the entire correction pipeline, at zero marginal cost. When the base model is strong, post-processing is unnecessary. When it's weak, upgrading the model is cheaper than building a correction pipeline.

The targeted correction uses [?word?] markers to tell the LLM exactly which words to focus on, reducing the chance of overcorrection on the parts that were already correct:

src/correction/llm_correction.pyMarked transcript for targeted correction
# Build transcript with markers around low-confidence words for i, w in enumerate(words): if i in flagged_set: marked_parts.append(f"[?{w['text']}?]") else: marked_parts.append(w["text"]) # Example output: # "मैंने [?सरियल?] की जांच [?कर्वा?] थी"
